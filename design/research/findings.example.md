## Research findings (example)

- **Goal**: Summarize research in a way that directly informs design decisions.

### Study summary

- **Method**: 6 remote usability sessions with existing users.
- **Focus**: Onboarding and first-project creation.

### Observed behaviors

- 4/6 participants hesitated on the project creation screen, unsure which option to pick.
- 5/6 participants missed the inline help link explaining project types.

### Underlying needs

- Users want a short, clear explanation of each project type before choosing.
- Users expect the default option to be safe and reversible.

### Opportunities

- Add concise descriptions directly under each project type label.
- Make one option clearly recommended (for example, “Best for most teams”) and explain why.
- Provide a small “Learn more about project types” link that opens a focused help panel instead of a separate docs site.
